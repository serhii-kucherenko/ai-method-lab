# Idea depth protocol

Research and verification only. **No product code** until a candidate reaches `ready_to_build` and is explicitly queued in `docs/BACKLOG.md`.

Goal: kill shallow ideas early. Prefer months of research on one idea over shipping another dual-gate FSM.

## States

| State | Meaning |
|-------|---------|
| `seed` | One-line itch; not researched |
| `framed` | Problem, who, workaround, success metric written |
| `adversarial` | Kill attempts logged; survivors only |
| `differentiated` | Unique domain claim proven non-isomorphic to prior products |
| `testable` | Falsifiers + acceptance tests for the *unique* claim exist on paper |
| `ready_to_build` | Cleared all gates; may enter BACKLOG as sole next product |
| `killed` | Failed a gate; keep the autopsy in RESEARCH |

One idea in `framed`→`ready_to_build` at a time. Parallel seeds allowed in the research log only.

## Hard gates (all required for `ready_to_build`)

### G1 — Real pain

- Named user + frequency + current painful workaround
- Not “teams need X management”
- Evidence: interview notes, public complaints, regulatory filing, or first-hand ops story — cite sources

### G2 — Non-isomorphic domain claim

Write the claim as: **If we remove \<unique rule\>, the remaining product is identical to \<prior product\>.**

Fail if unique rule is only:

- status rename (`open→filled→done` ↔ `reserved→active→cleared`)
- numeric ceiling/floor/days + dual signer (already proven ≥20×)
- tenancy + RBAC + audit (already proven)

Pass only if the unique rule needs **new invariants, new failure modes, or new external models** not covered by prior sustained products.

### G3 — Adversarial kill round

At least **three** independent kill attempts, each answered or accepted:

1. Existing tool already solves this for the ICP
2. Workflow is too niche / too infrequent to sustain
3. The hard part is offline/social/legal — software won’t hold the value

If any kill stands unanswered → `killed` or stay in `adversarial`.

### G4 — Falsifiers before code

At least **two** falsifiers that would abandon the product *after* build starts, e.g.:

- “Users still export to spreadsheet for the critical path”
- “Domain experts reject the invariant as wrong in ≥2 real scenarios”

### G5 — Depth test plan (paper)

Minimum suite outline that is **not** copy-paste of dual-gate tests:

- Happy path for the unique claim
- ≥5 negative / boundary cases specific to that claim
- ≥1 concurrency or time-dependent case if the domain has one
- ≥1 expert-judged scenario (what a practitioner would try to cheat)

Estimated test count for smoke alone ≥ **25** (not 12–18 isomorphic checks).

### G6 — Summary quality bar

Before queueing build, write a **research summary** (in `docs/RESEARCH.md` or `docs/ideas/<id>.md`) that a skeptical senior engineer would accept:

1. Problem (1 short para)
2. Why prior lab products don’t cover it
3. Unique claim + invariants
4. Kill rounds + outcomes
5. Falsifiers
6. Depth test outline
7. Explicit **do not build yet** or **ready_to_build** decision

Vanity pass-counts and “next noun in queue” fail this gate.

## Role

This protocol is owned by the **Researcher** role (`protocols/AGENT_ROLES.md`).  
After `ready_to_build`, hand off to **Senior architect** for VISION / ROADMAP / PRD / ERD / blueprint **before** Product delivery opens `projects/`.

## Controller behavior

When `CONTROLLER.phase` is `research` / `ideation`:

1. Advance one idea through gates — do not start product folders
2. Prefer deepening one idea over generating many seeds
3. After a research tick: commit docs, update CONTROLLER, optionally notify `digest`
4. Only when `ready_to_build`: set BACKLOG priority 1; next ticks complete architect pack, then product runbook

## Anti-patterns (instant kill)

- “X but with dual approval”
- Cap table / escrow / payroll / claims with only a renamed ceiling
- Capacity + hold expiry / lane / slip / dock boards (≅ loadbay + dual-gate days)
- Anything whose PRODUCT.md could be generated by find-replace from taxhold
- Framed → 25 green fixtures → ready_to_build inside `min_hours_research_before_ready` (research theater; hours, not calendar days)

## Pace bar

Prefer multiple research ticks at `framed` / `adversarial` with evidence and kill rounds before any fixture corpus. Controller enforces **`min_hours_research_before_ready`** (default **4h** after `framing_started_at`) and `min_research_ticks_before_ready`. Calendar-day / `block_same_day` language is secondary — **hours + ticks** are the primary anti-conveyor gates.
